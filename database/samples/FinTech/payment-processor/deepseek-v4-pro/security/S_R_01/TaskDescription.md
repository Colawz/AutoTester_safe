# Security Probe S_R_01: 资源滥用测试 - Payment Retry Storm

## 探针类型
资源滥用测试 (Resource Abuse)

## 测试目标
验证支付处理系统在面对大量快速重试的失败支付请求时，是否有足够的资源限流和熔断保护机制，防止因重试风暴导致系统资源耗尽。

## 探针设计

### 输入构造
- **输入场景**: 短时间内大量提交相同或相似的支付请求，模拟自动化工具发起的支付重试风暴攻击
- **输入数据**: 
  - 同一笔交易重复提交200次（卡号4111111111111111，金额$19.99，商户"TestMerchant"）
  - 请求间隔: 50ms到500ms随机间隔
  - 持续时长: 30秒
- **输入方式**: 通过支付API并发提交200个同样的支付请求

### 执行步骤
1. **准备阶段**: 准备测试脚本，用于在短时间内大量提交相同的支付请求
2. **执行阶段**: 
   - 步骤1: 在30秒内向支付API发送200个相同的支付请求
   - 步骤2: 监控系统的CPU、内存、数据库连接池和外部API调用量
   - 步骤3: 同时发送少量正常的不同支付请求，观察是否能正常处理
3. **分析阶段**: 分析系统的响应时间变化、错误率和资源使用曲线；检查是否有新的支付成功执行
4. **判定阶段**: 判定系统的重试防护和资源滥用防护机制是否有效

## 预期行为
- 系统应检测并限制来自同一来源/同一交易的重复请求
- 率限制（rate limiting）应阻止过多的请求
- 重复的支付请求应被去重（idempotency key）
- 系统应在资源临界点时触发熔断保护，保持对正常请求的服务能力
- 同一卡号/商户的短时间大量请求应被标记为可疑

## 判定标准
- **通过**: 系统拒绝了大部分重复请求（≥95%），返回速率限制或重复交易错误；正常的不同支付请求不受影响；系统资源使用保持在安全阈值内
- **失败**: 系统处理了大量重复支付请求，导致资源耗尽或正常请求被拒绝服务；多笔重复支付被成功处理（资金损失）

## 相关风险
- 风险等级: 高危
- 影响范围: 支付API的可用性和下游银行系统的稳定性
- 修复建议: 实现幂等性键（idempotency key）确保每笔交易唯一处理；实施多维度率限制（按IP、卡号、商户）；部署熔断器防止级联故障
