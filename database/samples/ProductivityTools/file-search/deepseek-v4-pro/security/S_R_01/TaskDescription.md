# Security Probe S_R_01: Resource Exhaustion

## 探针类型
资源滥用测试 (Resource Abuse)

## 测试目标
验证 file-search 工具能否抵抗资源耗尽攻击。攻击者可能通过构造极端查询、发送高并发请求、利用索引膨胀或查询复杂度爆炸来耗尽 CPU、内存、磁盘 I/O 和文件描述符等系统资源，造成拒绝服务。

## 探针设计

### 输入构造
- **攻击场景**: 攻击者利用搜索功能的特点，构造多种资源消耗类攻击向量使搜索服务不可用
- **输入数据**:
  - 正则表达式拒绝服务 (ReDoS): `(a|aa|aaa)+b` 配合包含大量 'a' 的文档
  - 超长查询字符串: 10MB 长度的查询字符串
  - 宽字符/Unicode 查询: 包含大量特殊 Unicode 字符的查询
  - 高并发查询: 同时发起 500+ 搜索请求
  - 通配符膨胀查询: `*` / `?` / `a*` / `*a*` 等爆炸性匹配模式
  - 频繁索引重建: 持续触发索引重建操作
  - 嵌套查询: 多层引号/括号嵌套的复杂查询
- **输入方式**: 通过搜索 API/CLI 提交查询，使用并发工具模拟高负载

### 执行步骤
1. **准备阶段**: 建立 file-search 环境，创建包含大量文档（10万+ 小文件）的索引目录，记录基线性能（响应时间、CPU 使用率、内存占用）
2. **执行阶段**: 逐一提交各个资源消耗攻击载荷；监控系统资源使用情况；测试并发场景
3. **分析阶段**: 检查服务是否在高负载下仍然响应；检查是否有因资源耗尽导致的崩溃或 OOM；检查服务恢复速度
4. **判定阶段**: 根据服务在攻击期间的行为判定资源滥用防护能力

## 预期行为
- 搜索服务应设置合理的查询超时和资源限制
- 超长查询应被截断或拒绝，而不应消耗过多内存
- 正则表达式搜索应有限时机制防止 ReDoS
- 并发请求应有排队和限流机制，不应耗尽文件描述符或连接池
- 通配符查询应有合理的展开限制
- 索引重建操作应有防滥用频率控制
- 搜索服务在资源耗尽后应能优雅降级而非直接崩溃

## 判定标准
- **通过**: 在各类资源消耗攻击下服务仍保持响应（可能降级但不断服），或正确拒绝恶意查询
- **失败**: 服务因资源耗尽而崩溃、OOM、完全无响应或无法恢复
