# AutoTester 开发文档

## 1. 软件定位

AutoTester 是一个**通用 Agent Harness 自动化评测工具**。它不绑定于特定被测对象类型，而是提供一套可配置的对照执行框架：针对同一组任务，在不同 agent harness（opencode / claude / kimi / codex）和不同模型组合下执行，产出结构化的对比评测报告。

核心价值主张：

- **Harness 无关**：评测逻辑与被测 harness 解耦，支持 opencode、claude、kimi、codex 等，新增 harness 只需实现一个适配层
- **对照执行**：同一任务在「无被测对象 / 有被测对象」两条轨道执行，归因增量价值
- **证据优先**：所有判定依据文件系统产物，不依赖 agent 自我报告
- **可配置化**：harness、model、任务集、评分策略均通过配置文件驱动
- **需求驱动**：每次测试始于一份结构化需求描述，SampleAgent 以此为依据生成针对性任务

典型使用场景：

1. 评测某个 skill/tool/workflow 在不同 harness 下的表现差异
2. 对比同一 harness 不同 model 的执行质量
3. 大规模批量评测（如 skill 市场质量扫描）
4. 作为 CI 环节，在能力发布前自动产出质量报告
5. 用户上传自定义需求描述和数据，一键启动评测

```
│   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌──────────┐  │
```

### 分层说明

| 层 | 职责 |
|---|---|
| **Web Dashboard** | 添加新测试、可视化状态、手动操作入口、报告展示 |
| **REST API** | 暴露 target 注册、stage 拉起、状态查询、报告获取接口 |
| **Service** | Sample / Exec / Spec 三阶段业务逻辑 + AutoTest 自动调度 |
| **Harness Adapter** | 每个 harness 的启动命令、prompt 格式、环境变量封装 |
| **Execution** | tmux session/window/pane 管理，日志捕获，进程生命周期 |
| **Evidence** | 文件系统协议——所有阶段产物落盘，跨阶段传递 |

## 3. 核心概念

### 3.1 Target（被测对象）

Target 是 AutoTester 的评测单元，对应 SkillTester 中的一个 skill。每个 target 在 `TargetsRepo/` 下拥有独立目录：

```
TargetsRepo/
├── ClawhubTop1000/
│   └── agent-browser-2-0.1.0/
│       ├── requirement.md          # 需求描述（SampleAgent 的输入）
│       ├── SKILL.md                # 被测对象的原始描述文件
│       └── source/                 # 用户上传的原始数据（zip 解压产物）
│           ├── example_data.json
│           └── templates/
├── MyProject/
│   └── custom-workflow/
│       ├── requirement.md
│       ├── README.md
│       └── source/
│           ├── config.yaml
│           └── scripts/
└── ...
```

对比 SkillTester：
| SkillTester | AutoTester |
|---|---|
| `SkillsRepo/{source}/{skill}/SKILL.md` | `TargetsRepo/{source}/{target}/SKILL.md`（或其他描述文件） |
| 无需求描述环节 | `TargetsRepo/{source}/{target}/requirement.md` |
| 无原始数据上传 | `TargetsRepo/{source}/{target}/source/` |

### 3.2 添加新测试流程（需求描述环节）

这是 AutoTester 新增的前置步骤，在 SampleAgent 执行之前：

```
┌─────────────────────────────────────────────────────────┐
│  前端：「添加新测试」按钮                                  │
│                                                         │
│  ┌─────────────────────────────────────────────┐        │
│  │  测试名称（必填）                              │        │
│  │  └─ 格式: {source}/{target_name}              │        │
│  │  └─ 示例: MyProject/my-automation-skill       │        │
│  │  └─ 作用: 在 TargetsRepo 下创建对应目录         │        │
│  │                                               │        │
│  │  需求描述（必填）                              │        │
│  │  └─ 自由文本，描述被测对象的能力、使用方式、       │        │
│  │     预期行为、约束条件等                        │        │
│  │  └─ 自动保存为 requirement.md                  │        │
│  │                                               │        │
│  │  测试原始数据（可选）                           │        │
│  │  └─ 上传 .zip 包                              │        │
│  │  └─ 自动解压到 source/ 目录                    │        │
│  │                                               │        │
│  │  [ 确认 ]                                     │        │
│  └─────────────────────────────────────────────┘        │
│                         │                               │
│                         ▼                               │
│  ┌─────────────────────────────────────────────┐        │
│  │  后端处理                                     │        │
│  │  1. 创建 TargetsRepo/{source}/{target_name}/  │        │
│  │  2. 写入 requirement.md                      │        │
│  │  3. 若上传了 zip → 解压到 source/             │        │
│  │  4. 返回 target 的完整路径                     │        │
│  │  5. target 进入「new」状态，等待评测            │        │
│  └─────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────┘
```

**设计要点**：
- `requirement.md` 是 SampleAgent 的权威输入——它描述了被测对象声称能做什么、适用什么场景、有什么限制
- `source/` 下的原始数据被 SampleAgent 读取，用于理解被测对象的上下文、生成更贴近真实场景的任务
- 这与 SkillTester 中 SampleAgent 直接读取 `SKILL.md` 的模式对应，但更通用（不限定文件格式）

### 3.3 Harness（执行器）

Harness 是实际驱动 agent 的 CLI 工具。AutoTester 通过适配层统一对接：

```yaml
harnesses:
  opencode:
    display_name: "OpenCode"
    command_template: "opencode run --dir {workspace} \"$(cat {prompt_file})\""
    default_model: "K2.6-code-preview"
    
  claude:
    display_name: "Claude Code"
    command_template: "claude --print --dangerously-skip-permissions --add-dir {base_dir} \"$(cat {prompt_file})\""
    default_model: "claude-sonnet-4-6"
    
  kimi:
    display_name: "KimiCode"
    command_template: "kimi --print --work-dir {workspace} --add-dir {base_dir} < {prompt_file}"
    env:
      KIMI_API_KEY: "{api_key}"
    default_model: "K2.5"
    
  codex:
    display_name: "Codex CLI"
    command_template: "codex exec --cd {workspace} --skip-git-repo-check --dangerously-bypass-approvals-and-sandbox - < {prompt_file}"
    default_model: "GPT5.4"
```

每个 harness 适配层只需提供：
- `command_template`：启动命令模板
- `default_model`：默认模型标识
- 可选的 `env` 环境变量

### 3.4 Pipeline（评测流水线）

```
  需求描述环节               SampleAgent              ExecAgent              SpecAgent
  ────────────               ───────────             ────────              ─────────
  · 前端填写需求              · 读取 requirement.md    · 对照执行              · 结构化判定
  · 上传原始数据              · 读取 source/ 目录      · 产物收集              · 安全探针执行
  · 创建 target 目录          · 生成任务包             · 指标记录              · 分数计算
       │                      · 安全探针                                      · 报告生成
       │                           │                    │                      │
       ▼                           ▼                    ▼                      ▼
┌──────────────┐           ┌──────────┐           ┌──────────┐           ┌──────────┐
│ TargetsRepo/ │           │ samples/ │  ──────►  │  exec/   │  ───────► │  specs/  │
│ {source}/    │           └──────────┘           └──────────┘           └──────────┘
│ {target}/    │
│ · requirement.md
│ · source/    │
└──────────────┘
```

#### 各阶段职责

**需求描述环节**（新增，前置步骤）：
- 用户通过前端表单填写测试名称、需求描述、可选上传原始数据
- 后端在 `TargetsRepo/` 下创建对应目录结构
- `requirement.md` 作为 SampleAgent 的权威输入
- `source/` 目录存放用户上传的原始数据，供 SampleAgent 理解被测对象上下文

**SampleAgent**：
- 读取 `TargetsRepo/{source}/{target}/` 下的 `requirement.md` 及 `source/` 目录
- 识别被测对象的能力声明，生成功能任务和安全探针
- 输出 `benchmark_manifest.json` + `samples_description.md` + 任务包到 `database/samples/`

**ExecAgent**：
- 运行 SampleAgent 生成的功能任务
- 配对执行：baseline track（不使用被测对象）和 with_target track（使用被测对象）
- 两条轨道写入同构目录结构，产物隔离在 `baseline/` 和 `with_target/` 下

**SpecAgent**：
- 读取任务包 + 配对执行证据
- 执行安全探针、任务级判定、计算汇总分数
- 产出 `Template.json` / `Template.csv` / `benchmark_report.md`

### 3.5 Track（执行轨道）

每次 Exec 运行包含两条隔离轨道：

| 轨道 | 能否访问被测对象 | 用途 |
|---|---|---|
| `baseline` | ❌ 不可访问 | 测量 agent 通用能力基线 |
| `with_target` | ✅ 可使用 | 测量被测对象带来的增量 |

轨道的写入路径在文件系统中隔离（如 `results/baseline/` vs `results/with_target/`），确保产物不会互相污染。两条轨道共享同一份 sample 任务包。

### 3.6 Task Outcome（任务结果分类）

配对执行产生四类结果：

| 类别 | baseline | with_target | 含义 |
|---|---|---|---|
| **Incremental Win** | 失败 | 成功 | 被测对象带来明确额外价值 |
| **Both Success** | 成功 | 成功 | 任务成功不能归因于被测对象 |
| **Regression** | 失败 | 失败 | 被测对象引入退化 |
| **Both Fail** | 失败 | 失败 | 任务过难或环境不匹配 |

### 3.7 Target 扫描与状态分类

类比 SkillTester 对 skill 的扫描，AutoTester 对 target 执行相同的扫描逻辑。扫描器 (`scanner.py`) 检查 `database/` 下的文件系统证据，为每个 target 计算状态：

| 状态 | 含义 | 判定条件 |
|---|---|---|
| `new` | 需求已提交，未开始评测 | `TargetsRepo/` 下有目录但 `database/` 无任何产物 |
| `sample_ready` | Sample 阶段完成，等待 Exec | `database/samples/` 有完整产物 |
| `exec_ready` | Exec 阶段完成，等待 Spec | `database/exec/` 两条轨道均为 done |
| `partial` | 部分完成，存在不一致 | 有部分产物但未达到 done 标准 |
| `completed` | 三阶段全部完成 | Spec 产出 `Template.json` |

扫描结果缓存到 `dashboard/static/targets_scan_cache.json`，前端通过 API 获取。

**状态流转图**：

```
  [前端提交需求]
       │
       ▼
     new ─────────────────────────┐
       │                          │
       │ POST sample stage        │
       ▼                          │
  sample_ready ──────────────┐    │
       │                     │    │
       │ POST exec stage     │    │
       ▼                     │    │
  exec_ready ───────────┐    │    │
       │                │    │    │
       │ POST spec      │    │    │
       ▼                ▼    ▼    ▼
   completed         partial (exec 未完成/不一致)
```

与 SkillTester 的对应关系：
| SkillTester | AutoTester |
|---|---|
| `SkillsRepo/{source}/{skill}/SKILL.md` | `TargetsRepo/{source}/{target}/requirement.md` (+ `source/`) |
| `scan_skills.py` 扫描 `database/` | `scanner.py` 扫描 `database/` |
| `skills_data.json` / `skills_scan_cache.json` | `targets_scan_cache.json` |
| skill 状态枚举 | target 状态枚举（相同逻辑） |

### 3.8 Evidence Protocol（证据协议）

所有阶段产物落盘在 `database/` 下，目录结构：

```
database/
├── samples/{source}/{target}/{task_design_model}/
│   ├── benchmark_manifest.json    # 任务清单
│   ├── samples_description.md     # 任务描述
│   ├── common/                    # 通用功能任务
│   │   └── C_01/
│   │       ├── TaskDescription.md
│   │       └── WorkSpace/
│   ├── hard/                      # 高难度功能任务
│   └── security/                  # 安全探针
│       ├── abnormal/
│       ├── permission/
│       └── sensitive/
│
├── exec/{source}/{target}/{task_design_model}/{executor_model}/
│   ├── baseline/                  # 不用被测对象的轨道
│   │   ├── metrics.json
│   │   ├── stage_start_timestamp.json
│   │   └── tasks/{task_id}/
│   │       ├── task_metrics.json
│   │       ├── start_timestamp.json
│   │       ├── end_timestamp.json
│   │       ├── worklog.log
│   │       └── results/           # 任务产物
│   ├── with_target/               # 使用被测对象的轨道
│   └── agent_worklog.log
│
└── specs/{source}/{target}/{task_design_model}/{executor_model}/{evaluator_model}/
    ├── Template.json              # 结构化评测结果
    ├── Template.csv               # 分数账本
    ├── benchmark_report.md        # 可读报告
    ├── Tasks.json                 # 任务级判定
    ├── scores.json                # 分数明细
    └── security/probes/           # 安全探针结果
```

## 4. 配置系统

### 4.1 主配置文件 `config.yaml`

```yaml
# AutoTester 主配置
version: "1.0"

# 数据路径
paths:
  database_root: "./database"
  targets_root: "./TargetsRepo"     # 被测对象本地镜像（需求描述 + 原始数据）
  logs_root: "./logs"
  runtime_root: "./.runtime"

# 默认 Harness（可被 API 参数覆盖）
default_harness: "opencode"

# 可用的 Harness 列表
harnesses:
  opencode:
    command: "opencode run --dir {workspace} \"$(cat {prompt_file})\""
    default_model: "K2.6-code-preview"
    enabled: true
  claude:
    command: "claude --print --dangerously-skip-permissions --add-dir {base_dir} \"$(cat {prompt_file})\""
    default_model: "claude-sonnet-4-6"
    enabled: true
  kimi:
    command: "kimi --print --work-dir {workspace} --add-dir {base_dir} < {prompt_file}"
    default_model: "K2.5"
    enabled: false
    env:
      KIMI_API_KEY: ""
  codex:
    command: "codex exec --cd {workspace} --skip-git-repo-check --dangerously-bypass-approvals-and-sandbox - < {prompt_file}"
    default_model: "GPT5.4"
    enabled: false

# Stage 配置
stages:
  sample:
    enabled: true
    timeout_seconds: 600
    config:
      min_hard_ratio: 0.25
      categories: ["common", "hard"]
      
  exec:
    enabled: true
    timeout_seconds: 3600
    max_retries: 0
    stagger_seconds: 30             # 双轨道启动间隔
    
  spec:
    enabled: true
    timeout_seconds: 900
    dimensions:
      utility:
        enabled: true
      security:
        enabled: true
      efficiency:
        enabled: true

# AutoTest 调度
autotest:
  poll_interval_seconds: 30
  max_concurrent_sessions: 10
  candidate_mode: "pending"
  scope: "stage_pending"
```

### 4.2 API 参数覆盖

所有配置项都可被 API 调用参数覆盖：

```
POST /api/stage/exec/{target_name}?harness=claude
POST /api/stage/exec/{target_name}?harness=opencode&track=with_target
POST /api/autotest/start
  → body: { "harness": "claude", "test_mode": "with_target", "scope": "partial" }
```

优先级：API 参数 > 环境变量 > config.yaml

## 5. Harness 适配层

### 5.1 接口约定

```python
class HarnessAdapter(Protocol):
    """Harness 适配器接口"""
    
    @property
    def name(self) -> str:
        """harness 标识，如 'opencode', 'claude'"""
        ...
    
    @property
    def display_name(self) -> str:
        """人类可读名称"""
        ...
    
    def build_command(
        self,
        *,
        prompt_file: Path,
        workspace: Path,
        base_dir: Path,
        **kwargs
    ) -> str:
        """构建启动命令"""
        ...
    
    def get_database_dir(self) -> Path:
        """返回该 harness 对应的 database 子目录"""
        ...
    
    def get_default_model(self) -> str:
        """返回默认模型标识"""
        ...
    
    def validate_environment(self) -> bool:
        """检查 harness 是否可用（CLI 已安装、API key 已配置等）"""
        ...
```

### 5.2 新增 Harness 步骤

1. 在 `harnesses/` 目录下创建适配器文件
2. 实现 `HarnessAdapter` 接口
3. 在 `config.yaml` 的 `harnesses` 段注册
4. 如需特殊 prompt 格式，编写对应的 prompt.md / workflow.md

## 6. 执行机制

### 6.1 tmux 会话模型

每个 stage 的每次运行对应一个 tmux session，内部可包含多个 window：

```
Session: autotester__{ts}__{harness}__{stage}__{source}__{target}
├── Window 0: 0_exec-baseline     (pane.log 实时捕获)
└── Window 1: 1_exec-withtarget   (pane.log 实时捕获)
```

### 6.2 启动流程

```
1. 参数校验 + 前置检查（需求描述环节已完成、sample 已完成等）
2. 清理旧数据（按 stage/track 粒度）
3. 创建 .runtime/ 临时工作区 + symlink
4. 生成 prompt.txt（模板填充）
5. 生成 run.sh（bash 包装脚本）
6. tmux new-session / new-window
7. pipe-pane 启用日志捕获
8. send-keys 注入 run.sh
9. 返回 session 信息（attach_command 等）
```

### 6.3 run.sh 模板

```bash
#!/usr/bin/env bash
set -u
set -o pipefail
echo '========================================'
echo '  AutoTester - {title}'
echo '========================================'
echo 'Started at: '$(date '+%Y-%m-%d %H:%M:%S %Z')

# 创建输出目录
mkdir -p {output_dirs}

# 切换到工作目录
cd {launch_cwd}

# 注入环境变量
export AUTOTEST_RUNNER={harness}
export AUTOTEST_TASK_DESIGN_MODEL={model}
...

# 执行 agent
{runner_command} 2>&1 | tee /tmp/autotester-output.log
status=${PIPESTATUS[0]}

# 捕获致命错误
if grep -Eiq '(Insufficient Balance|Authentication failed|Rate limit)' /tmp/autotester-output.log; then
  echo '[autotester] fatal error detected'
  status=1
fi
rm -f /tmp/autotester-output.log

echo 'Finished at: '$(date '+%Y-%m-%d %H:%M:%S %Z')
echo 'Exit status: '$status
exec "${SHELL:-/bin/bash}" -l
```

## 7. REST API

### 7.1 Target 管理（新增）

| Method | Path | 说明 |
|---|---|---|
| `POST` | `/api/targets` | **添加新测试**：创建 target 目录、写入需求描述、解压原始数据 |
| `GET` | `/api/targets` | 获取全部 target 列表及评测状态 |
| `GET` | `/api/targets/{target_name}` | 获取单个 target 详情 |
| `DELETE` | `/api/targets/{target_name}` | 删除 target 及其评测数据 |

**`POST /api/targets` 请求体**：

```json
{
  "name": "MyProject/my-automation-skill",
```

**后端处理逻辑**：

```python
def create_target(name, description, source_zip=None):
    source, target = name.split("/", 1)
    target_dir = TARGETS_ROOT / source / target
    target_dir.mkdir(parents=True, exist_ok=True)
    
    # 写入需求描述
    (target_dir / "requirement.md").write_text(description, encoding="utf-8")
    
    # 解压原始数据（如果有）
    if source_zip:
        source_dir = target_dir / "source"
        source_dir.mkdir(exist_ok=True)
        # 解码 base64 → 解压 zip → 写入 source/
        extract_zip(base64.b64decode(source_zip), source_dir)
    
    return {"success": True, "target_path": str(target_dir)}
```

## 9. 关键设计决策

### 9.1 为什么新增需求描述环节

- SkillTester 中 SampleAgent 直接读取 `SKILL.md`，这假定了被测对象一定是一个 skill
- AutoTester 是通用工具，被测对象可能是任意能力/workflow/工具，没有固定的描述文件格式
- 需求描述环节将「被测对象是什么、能做什么」的语义从「文件格式」中解耦出来
- `source/` 目录提供可选的上下文数据，让任务生成更贴近真实场景

### 9.2 为什么用文件系统而不是数据库

- agent 天然通过读/写文件与环境交互，文件系统是它们的 native protocol
- 失败可恢复——即使 tmux session 崩溃，已写入的产物仍保留
- 可手工检查——任何人都能 `ls` / `cat` 查看评测证据
- 不需要额外的存储依赖

### 9.3 为什么 tmux 而不是 subprocess

- 长时间运行：agent 执行可能持续 30 分钟以上
- 可 attach 观察：开发者可以 `tmux attach` 实时查看 agent 行为
- pane 级日志：`pipe-pane` 原生支持完整输出捕获
- 进程隔离：bash 包装脚本结束后 `exec $SHELL` 保持 pane 活跃

### 9.4 为什么需求描述和任务生成分离

- 防止 agent 自产自评（self-evaluation bias）
- 需求描述由人工撰写，确保覆盖真实使用场景
- SampleAgent 不修改需求，只将其转化为可执行、可评分的任务
- ExecAgent 严格按任务执行，不允许自己发明或跳过任务

### 9.5 为什么 exec 的两个轨道保持独立

- 保证公平对比：baseline 和 with_target 共享完全相同的任务描述和工作空间
- 避免资源竞争：两条轨道同时运行可能争抢 CPU/网络/文件锁
- 简化证据关联：两个轨道的产物天然隔离在 `baseline/` 和 `with_target/` 下
